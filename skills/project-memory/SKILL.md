# 项目记忆

## 作用

维护两个项目级记忆文件，让未来的 Codex 会话可以快速理解项目当前进度，以及接下来应该做什么。

这个 skill 不负责完整的上下文恢复。它只负责初始化、读取、总结和维护项目记忆文件；后续可以由另一个恢复上下文的 skill 基于这两个文件继续工作。

## 文件

在项目根目录使用这两个文件：

- `PROJECT_PROGRESS.md`：记录项目当前状态。
- `PROJECT_TODO.md`：只记录尚未完成的后续工作、开放问题、阻塞点和待跟进事项。

新项目记忆不要再使用 `PROJECT_STATUS.md`，除非用户明确要求使用旧格式。

## 工作流程

1. 根据当前工作目录和附近的项目标记确定项目根目录，例如 `.git`、`AGENTS.md`、包管理文件，或用户提供的路径。
2. 检查项目根目录是否存在 `PROJECT_PROGRESS.md` 和 `PROJECT_TODO.md`。
3. 如果任一文件不存在，就按下面模板创建；已存在的文件不要覆盖。
4. 确认两个文件存在后，读取它们。
5. 向用户总结项目记忆：
   - 当前阶段和状态
   - 已完成内容
   - 重要决策
   - 立即要做的下一步
   - 阻塞点或需要确认的事项
6. 完成有意义的工作后，更新对应文件：
   - 进度、决策、已完成内容写入 `PROJECT_PROGRESS.md`
   - 下一步、阻塞、待确认事项写入 `PROJECT_TODO.md`

## 初始化模板

### PROJECT_PROGRESS.md

```markdown
# 项目进度

## 基本信息
- 项目名称：
- 当前阶段：需求 / 设计 / 开发 / 测试 / 维护
- 最后更新：YYYY-MM-DD

## 当前状态
-

## 已完成
-

## 关键决策
- YYYY-MM-DD:

## 技术 / 结构备注
-

## 最近一次进展
- YYYY-MM-DD:
```

### PROJECT_TODO.md

```markdown
# 项目待办

## 下一步
- [ ]

## 进行中
- [ ]

## 待确认
- [ ]

## 阻塞
- [ ]

## 后续可做
- [ ]
```

## 维护规则

- 内容保持简洁、事实化，不写空泛总结。
- 优先更新已有条目，避免重复记录相似事项。
- 关键决策和最近进展要带日期。
- `PROJECT_TODO.md` 只能保留未完成事项；不要长期保留已勾选的完成项。
- 任务完成后，把结果写入 `PROJECT_PROGRESS.md`，并必须从 `PROJECT_TODO.md` 删除对应事项。
- 如果发现的问题不属于当前任务范围，先向用户说明问题、影响和建议选项，由用户决定是否立即处理、记录到 `PROJECT_TODO.md` 或忽略。
- 在长期项目暂停前，检查这两个文件是否反映了最新工作状态。
- 文档、计划、总结、待办等面向用户或项目维护的内容默认使用中文；代码、命令、文件路径、API 名称和专有名词保持原文。

## 总结格式

向用户汇报项目记忆时，保持简短：

```markdown
当前状态：

下一步：

阻塞 / 待确认：
```

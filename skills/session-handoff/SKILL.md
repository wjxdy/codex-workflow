# 会话交接

## 作用

把当前会话的开发现场详细记录到项目根目录的 `SESSION_HANDOFF.md`，让用户清空上下文或重开会话后，可以通过另一个恢复类 skill 快速继续当前任务。

这个 skill 只负责“写现场交接文档”。不要主动清空上下文，不要开始恢复流程，不要继续开发新功能，也不要擅自提交。

## 文件

在项目根目录维护：

- `SESSION_HANDOFF.md`：当前会话的详细交接记录，面向下一次恢复上下文使用。

相关但职责不同的文件：

- `PROJECT_PROGRESS.md`：长期项目状态。
- `PROJECT_TODO.md`：未完成待办，只保留尚未完成事项。

`SESSION_HANDOFF.md` 可以很详细、很具体；不要把这些临时现场细节塞进 `PROJECT_PROGRESS.md` 或 `PROJECT_TODO.md`。

## 工作流程

1. 确定项目根目录。优先根据 `.git`、`AGENTS.md`、包管理文件、用户提供路径和当前工作目录判断。
2. 读取项目记忆文件：
   - 若存在 `PROJECT_PROGRESS.md`，读取它。
   - 若存在 `PROJECT_TODO.md`，读取它。
   - 不存在时不要自动初始化；只在交接文档里注明缺失。
3. 收集当前工作区状态：
   - 当前分支
   - 最近提交
   - `git status --short`
   - 已修改、新增、删除、未跟踪文件
4. 梳理当前会话上下文：
   - 用户原始目标
   - 当前正在做的任务
   - 已经完成的事
   - 正在编辑或需要继续编辑的文件
   - 未完成事项
   - 已运行验证命令及结果
   - 阻塞、风险、待确认点
5. 写入或覆盖项目根目录的 `SESSION_HANDOFF.md`。
6. 如发现 `PROJECT_TODO.md` 里缺少明确的下一步，可以先询问用户是否同步；不要擅自把大量交接细节写入 `PROJECT_TODO.md`。
7. 完成后告诉用户：
   - `SESSION_HANDOFF.md` 的路径
   - 交接文档记录了哪些重点
   - 下一次可调用恢复类 skill 读取它继续工作

## 记录原则

- 默认使用中文。
- 具体胜过概括。要写到下一次会话能直接继续，不需要重新猜。
- 文件路径必须具体，尽量写绝对路径或项目相对路径。
- 对每个关键文件说明：状态、用途、已做改动、下一步要做什么。
- 验证状态必须诚实记录：跑过什么、结果是什么、什么没跑。
- 不要把“可能已经完成”写成“已经完成”；不确定就标明不确定。
- 不要删除或覆盖用户未要求删除的项目记忆文件。
- 不要把已完成待办保留在 `PROJECT_TODO.md`；完成结果应进 `PROJECT_PROGRESS.md`，待办项应删除。

# 恢复上下文

## 作用

读取项目记忆、会话交接文件和 git 状态，帮助 Codex 理解上个会话在做什么、做到哪里、接下来可能要做什么。

这个 skill 只负责“恢复上下文认知”，不代表授权继续开发。

## 硬规则

- 调用本 skill 只代表恢复上下文，不代表授权继续开发。
- 除非用户在同一句话里明确说“恢复后继续开发”或“恢复后直接执行下一步”，否则恢复完成后必须停止，等待用户下一条指令。
- 不要自动修改文件。
- 不要自动运行大型测试、构建或启动服务。
- 不要自动提交、推送、清理工作区或执行破坏性命令。
- 不要把 `SESSION_HANDOFF.md` 里的“下一步执行计划”当成执行许可。

## 需要读取的信息

在项目根目录优先读取：

- `PROJECT_PROGRESS.md`：长期项目进度、当前阶段、关键决策。
- `PROJECT_TODO.md`：尚未完成的待办、阻塞、待确认事项。
- `SESSION_HANDOFF.md`：上个会话的详细现场记录，是恢复上下文时最重要的文件。

还需要读取 git 信息：

- 当前分支：`git branch --show-current`
- 最近提交：`git log --oneline -n 5`
- 工作区状态：`git status --short`
- 必要时读取变更概览：`git diff --stat`

不要因为缺少上述文件就自动初始化或创建文件；只在恢复摘要里说明缺失。

## 工作流程

1. 确定项目根目录。优先根据 `.git`、`AGENTS.md`、包管理文件、用户提供路径和当前工作目录判断。
2. 读取 `PROJECT_PROGRESS.md`、`PROJECT_TODO.md`、`SESSION_HANDOFF.md`。若某个文件缺失，记录为缺失。
3. 读取 git 当前分支、最近提交、工作区状态。
4. 对照 `SESSION_HANDOFF.md` 检查一致性：
   - 当前分支是否和交接记录一致。
   - 最近提交是否和交接记录一致。
   - 关键文件是否仍存在。
   - 当前工作区改动是否能和交接记录对应。
5. 输出恢复摘要：
   - 项目背景
   - 上个会话的任务
   - 当前进度
   - 关键文件
   - 下一步建议
   - 风险、阻塞和待确认
   - 一致性检查结果
6. 停止并等待用户下一条指令。不要继续开发。

## 一致性检查规则

如果发现以下情况，必须在恢复摘要里标为“需要确认”：

- `SESSION_HANDOFF.md` 缺失。
- 当前分支和交接文档记录不一致。
- 最近提交和交接文档记录不一致，且看起来有新提交。
- 交接文档提到的关键文件不存在。
- `git status --short` 中存在大量交接文档没有解释的改动。
- `PROJECT_TODO.md` 和 `SESSION_HANDOFF.md` 的下一步明显冲突。

如果只是正常的未完成工作区改动，并且能和交接文档对应，可以标为“可继续”，但仍然等待用户确认是否继续。

```markdown
|---|---|---|---|

## 下一步建议

## 风险 / 阻塞 / 待确认
-

-

- 继续开发
- 先解释当前状态
- 先检查某个文件
- 暂停，不继续
```

## 如果用户同时要求继续

如果用户明确说“恢复后继续开发”，先完成恢复摘要，再按 AGENTS.md 的任务分级和开发流程继续。  
如果恢复过程中发现一致性问题或阻塞，先暂停并让用户决定，不要强行继续。

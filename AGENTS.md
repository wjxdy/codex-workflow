# Codex 使用说明

## 🌐 回复与文档语言

- 默认使用中文回答。
- 默认使用中文编写面向用户或项目维护的文档、计划、总结、待办、评审说明和项目记忆文件；除非用户明确要求其他语言。
- 代码、命令、文件路径、API 名称和专有名词保持原文。

## 🧭 交互分型（先判断再决定流程）

| 类型 | 触发词 | 处理方式 |
|---|---|---|
| **咨询** | "为什么"、"解释"、"看下"、"怎么理解"、"帮我分析" | 直接答，不走流程，不建任务列表 |
| **改动** | "做"、"实现"、"修"、"加"、"改" | 按下面开发流程走 |

---

## ⚠️ 法则一：开发流程（按任务级别分层）

### 任务分级与流程

| 级别 | 定义 | 流程 | 必用 Skill |
|---|---|---|---|
| **P0** | 新模块、新 API、≥3 文件、跨层改动 | 方案 → **用户确认** → 实现 → review → 提交 | brainstorming → writing-plans → subagent-driven-development |
| **P1** | 单页/单组件/单接口，有明确交互 | 简短方案 → **用户确认**（Auto Mode 跳过此步）→ 实现 → 自审 → 提交 | brainstorming（简）→ writing-plans（简）|
| **P2** | 改常量/样式/typo/配置、< 10 行 | 直接改 → 验证 → 提交 | 按需，不强制 |

**降级信号**：用户说"直接改"、"小改一下"、"快速修复"，或改动一眼 < 10 行 → P2 处理。

### 通用原则

- 用户说"可以"、"好的"、"确认"后才进入下一步（仅**手动模式** P0/P1 方案阶段；**Auto Mode 以下方「Auto Mode 豁免」为准**，不等确认）
- 文档与代码一起改，一起审
- 验证失败禁止进入下一阶段或声称完成

### 验证要求（无例外）

| 类型 | 要求 |
|---|---|
| 后端 | 编译/启动后用 API/单元/集成测试验证目标功能 |
| 前端 | build 通过后浏览器验证交互，或跑 Playwright E2E |
| 全链路 | 前后端一起启动，走完整用户路径 |

### Playwright 自动验证（仅 P0/P1 且涉及真实 UI 交互时启用）

- **启用**：改动明确包含路由、点击、输入等交互
- **不启用**：纯样式/配置/常量调整，或后端-only
- 机制：推断路由与步骤 → 浏览器执行 → 断言；失败标 `blocked` + 截图 + 暂停
- 服务自动拉起，端口占用即复用，单步超时 45s

### Code Review

| 场景 | 方式 |
|---|---|
| P0 | 调 `superpowers:requesting-code-review`（子 agent 审） |
| P1/P2 | 自审 checklist（类型一致、边界、测试覆盖）+ 回复中说明"我检查了 X" |
| 紧急修复 | 主动指出"改了 X，可能影响 Y"，由用户决定是否深入 |

### Auto Mode 豁免

- **P0**：方案阶段仍需用户确认；实现阶段自主执行，完成后汇报
- **P1/P2**：全程自主，**仅以下情况暂停询问**：
  1. 验证/测试失败且原因不明（hook 会在测试/构建命令失败时强制注入提醒）
  2. 破坏性操作（删库、删分支、force-push、清数据）—— 必须先口头询问并说明风险；用户明确同意后由 Codex 执行
  3. 用户中途需求与已确认方案结构性冲突
  4. 安全合规风险（暴露密钥、CI/CD 权限）—— 必须先说明风险并获得用户确认

## 法则二：通用 Skill 触发（流程外按需使用）

| 场景 | Skill |
|---|---|
| 遇到 bug/测试失败 | `systematic-debugging` |
| 声称完成前 | `verification-before-completion` |
| 需要隔离工作区 | `using-git-worktrees` |
| 2+ 独立任务并行 | `dispatching-parallel-agents` |

### 探索路由（强制）

| 信号 | 走向 |
|---|---|
| "X 在哪"、"找一下 Y"、"有没有 Z"、"哪个文件" | `Explore` subagent，**禁止**主线程直接 grep |
| "为什么 X 这么写"、"这块逻辑是啥" | 先 `Explore` 拿上下文，再主线程解释 |
| 已知确切路径/符号 | 主线程直接 Read/grep，不走 Explore |

## 法则三：项目记忆文件

### 建议询问初始化的场景

- 项目预计持续 3 次以上会话
- 有明确阶段性目标（MVP/上线/迭代周期）
- 用户显式要求
- 用户主动调用 `project-memory`

以上场景只表示应该主动询问用户是否初始化项目记忆文件；不要自行创建。一次性脚本、实验 demo、小工具不需要询问。

### 启用后规则

- 位置：项目根 `PROJECT_PROGRESS.md` 与 `PROJECT_TODO.md`
- `PROJECT_PROGRESS.md`：记录项目当前阶段、已完成内容、关键决策、技术/结构备注、最近进展
- `PROJECT_TODO.md`：记录下一步、进行中事项、待确认、阻塞、后续可做
- 会话开始时主动读取两个项目记忆文件；若两个文件不存在或缺任一文件，主动询问用户是否调用 `project-memory` skill 初始化，不要自行按条件创建
- 更新时机：阶段变更、需求增减、阻塞解决（commit 后 hook 自动提醒；会话结束前若 commit 后未更新，Stop hook 会再次提醒）
- 发现不属于当前任务范围的问题时，先向用户说明问题、影响和建议选项，由用户决定：立即作为当前任务的前置或子任务处理、记录到 `PROJECT_TODO.md` 本次不处理，或忽略；若该问题会导致当前验证无法继续，明确说明它是当前阻塞，但仍由用户确认如何处理
- 与功能 commit 一起提交

### 模板来源

两个项目记忆文件的模板由 `project-memory` skill 统一维护；`AGENTS.md` 只定义触发与维护规则，不重复写模板。

### 相关 Skill

- `project-memory`：初始化或维护 `PROJECT_PROGRESS.md` 与 `PROJECT_TODO.md`。
- `session-handoff`：当上下文过满、准备重开会话或暂停开发时，写入 `SESSION_HANDOFF.md`，详细记录当前任务、关键文件、验证状态和下一步计划。
- `restore-context`：重开会话后读取 `PROJECT_PROGRESS.md`、`PROJECT_TODO.md`、`SESSION_HANDOFF.md` 和 git 状态，只恢复上下文认知；除非用户明确要求继续开发，否则不得自动修改文件。

## 法则四：Git 提交与推送

- **频繁本地 commit**：每个独立功能点测试通过后立即 commit，单 commit 单意图
- **独占 feature 分支自动 push**：仅限**独占**的 feature 分支；**共享分支**（多人协作同一 feature）需先询问
- **合入主分支**：push/merge 到 main/master、force-push 到 main/master、`--no-verify`/`--no-gpg-sign` 属于高风险操作，必须先口头询问并说明将执行的命令；用户明确同意后由 Codex 执行。若工具、hook 或权限拦截，再向用户说明阻塞原因
- **创建并合并 PR**：仍需先口头询问，列出变更概要后执行
- **不积压**：单次会话累计 ≥3 个未推送 commit 时主动提醒推送

## 法则五：危险操作与敏感文件

### 危险操作原则

危险操作默认暂停，必须先向用户说明：

- 准备执行什么命令
- 为什么需要执行
- 可能影响什么范围
- 是否有更保守的替代方案

用户明确同意后，由 Codex 执行；不要要求用户自己去终端执行。若工具、hook 或权限拦截，再向用户说明阻塞原因。

### 需要先询问的操作

- `git push --force` / `--force-with-lease`
- `git push` / `git merge` 到 `main` / `master`
- `git commit --no-verify` / `--no-gpg-sign`
- `git reset --hard` 涉及未提交工作
- `git clean -fd` / `git checkout .` / `git restore .` 整目录回滚
- `git branch -D` 删除未合并分支
- `rm -rf` 涉及版本控制目录
- `gh pr merge` / `gh pr close`
- 写入 / 修改 `~/.gitconfig`、CI/CD 配置文件

### 敏感文件

`.env`、凭据、SSH 私钥、证书、云配置等敏感文件，默认不直接写入真实秘密值。

如果用户明确要求修改，需要先说明风险并获得确认；能通过占位符、示例值、模板或明确的编辑建议解决时，优先不用真实敏感内容。
